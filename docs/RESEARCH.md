# Market and competitor analysis

## Research method

The 2026-06-22 ecosystem snapshot in [PROJECT-TITAN.md](PROJECT-TITAN.md) records public GitHub discovery counts. Counts are directional and time-stamped; they are not growth rates. Repeat the collection monthly, retain raw links and dates, and calculate growth only from the stored series. Product claims need a primary source (documentation, pricing, changelog, or repository) and a retrieval date.

## What established repositories reveal

| Category | Users love | Common failure | Titan response |
| --- | --- | --- | --- |
| Prompt collections | Immediate copy/paste utility and breadth | Generic, stale, untested entries | Small decision workflows with evaluation gates |
| Rules and memories | Local context and consistent conventions | Vendor lock-in; weak portability | Plain Markdown source plus thin adapters |
| MCP directories | Tool discovery and community curation | Discovery is not a reliable workflow | Tool-agnostic procedures with evidence requirements |
| Agent frameworks | Automation and composability | High setup cost; opaque failure modes | Start with a human-verifiable decision record |
| Model-specific skills | Familiar UX and fast adoption | Model churn and marketing-first content | Durable engineering problems as the organizing principle |

## Why projects stall below meaningful adoption

They often offer interchangeable content, have no reproducible proof, make installation unclear, accept unmaintained submissions, or optimize a launch spike instead of repeat use. Large repositories tend to pair discoverability with a dependable contributor loop, concrete artifacts, and strong naming/distribution. Star count is not product-market fit.

## Underserved need

Teams need procedures that produce decisions they can inspect: evidence, unknowns, trade-offs, verification, ownership, and a next review. High-stakes work—security, architecture, data migration, and spend—benefits most because a fluent answer alone is not enough.

## Trend hypotheses to test

1. Evaluation datasets and traceable provenance become stronger differentiators than prompt text.
2. Model-agnostic authoring wins as agent surfaces proliferate.
3. Skills converge with policy, tool permissions, and structured outputs; this repository should remain compatible without bundling unsafe permissions.
4. Teams will value maintenance signals—owners, review dates, deprecations—as highly as novelty.
